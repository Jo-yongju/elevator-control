# STM32 기반 엘리베이터 제어 시스템

STM32와 FS90R 연속회전 서보모터로 구현한 5층 엘리베이터 제어 시스템입니다.
초기 단일 파일 구조를 상태 머신 기반으로 재설계하고, SCAN 스케줄링과
센서 기반 속도 프로파일링, 안전 로직을 추가했습니다.

## 하드웨어 구성

- MCU: STM32 (HAL)
- 구동: FS90R 연속회전 서보모터 (PWM 제어)
- 위치 감지: 층별 홀 센서 5개
- 입력: 4x4 키패드
- 표시: 7-세그먼트 디스플레이

## 하드웨어 한계와 설계 방향

FS90R은 엔코더가 없어 **현재 속도나 위치를 측정할 수 없습니다.**
따라서 오차를 피드백받아 보정하는 PID 제어는 적용할 수 없습니다.
홀 센서도 층마다 한 개씩이라 층과 층 사이의 연속 위치는 알 수 없습니다.

이 한계를 전제로, 측정이 아닌 **이벤트(홀 센서 감지)를 트리거**로
동작을 전환하는 방식으로 설계했습니다.

## 소프트웨어 구조

하드웨어 제어와 판단 로직을 레이어로 분리했습니다.

| 모듈 | 역할 |
|------|------|
| `motor` | PWM 듀티로 서보 속도/방향 제어 (하드웨어 추상화) |
| `keypad` | 4x4 키패드 매트릭스 스캔 |
| `display` | 7-세그먼트 숫자 표시 |
| `hall_sensor` | 홀 센서로 현재 층 감지 (디바운싱 포함) |
| `elevator` | 상태 머신 + SCAN 스케줄링 (핵심 판단 로직) |
| `main` | 초기화 및 메인 루프 |

상위 로직(`elevator`)은 핀 번호나 PWM 값 같은 하드웨어 세부사항을
알지 못하며, `Motor_SetDir(MOTOR_UP)`처럼 의미 단위 명령만 사용합니다.

## 핵심 구현

### 1. 상태 머신 기반 비블로킹 구조

초기 코드는 이동 처리가 중첩 `while` 루프 안에 갇혀 있어,
이동 중에는 다른 입력을 받지 못했습니다. 이를 5개 상태
(`IDLE`, `MOVING`, `DOOR_OPEN`, `DOOR_CLOSING`, `FAULT`)의
상태 머신으로 재설계했습니다.

`Elevator_Update()`는 매 루프에서 현재 상태에 해당하는 처리를
한 단계만 수행하고 즉시 반환합니다. 블로킹 지점이 없어 어떤
상황에서도 입력 응답성을 유지합니다.

### 2. SCAN 스케줄링

초기에는 호출이 들어온 순서대로 처리(FIFO)해, 진행 방향과 무관하게
오래된 호출을 먼저 처리하는 비효율이 있었습니다. 진행 방향을 유지한 채
같은 방향의 호출을 지나가며 함께 처리하고, 그 방향에 호출이 없을 때만
방향을 전환하는 SCAN 방식으로 변경했습니다.

### 3. 이벤트 기반 속도 프로파일링

속도 측정이 불가능한 FS90R의 한계 안에서 충격을 줄이기 위해,
세 구간으로 속도를 나눴습니다.

- 출발: 시간 기반으로 PWM을 단계적으로 높임 (오픈루프 가속)
- 이동: 정상 속도 유지
- 목적지 한 층 전(홀 센서 감지): 저속으로 전환
- 목적지 도달: 정지

목적지 한 층 전 감속은 홀 센서 이벤트를 트리거로 사용합니다.
속도를 측정해 보정하는 PID와 달리, 미리 정해둔 저속 펄스로
전환하는 방식입니다.

### 4. 안전 로직

- **홀 센서 디바운싱**: 센서 신호가 짧게 흔들리는 문제가 있어,
  일정 시간(20ms) 동안 같은 상태가 유지될 때만 감지로 인정합니다.
- **도어 인터락**: 문이 열린 상태에서는 모터를 강제로 정지로 묶어,
  문이 열린 채 이동하는 상황을 차단합니다.
- **타임아웃 비상 정지**: 이동 시작 후 일정 시간 안에 다음 층을
  감지하지 못하면 고장으로 판단해 `FAULT` 상태로 진입, 모터를 멈춥니다.

## 제어 방식에 대한 정리

본 프로젝트의 속도 제어는 **PID가 아닙니다.** FS90R이 피드백
수단을 제공하지 않기 때문입니다. 대신 하드웨어 한계를 인식하고
홀 센서 이벤트 기반의 속도 프로파일링과 오픈루프 가감속으로
구현했습니다.
